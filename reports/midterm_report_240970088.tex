\section{1 Introduction}

\subsection{1.1\quad Overview of the Project}

Sculpt is a real-time collaborative visual feedback platform designed to streamline the design review and iteration process for creative teams. The web-based application enables designers, developers, project managers, and stakeholders to upload visual assets, annotate them directly on a shared canvas, and engage in structured threaded discussions --- all within a unified browser-based interface.

In modern product development workflows, visual feedback is often scattered across email threads, messaging applications, and standalone annotation tools. This fragmentation leads to miscommunication, lost context, and delayed iteration cycles. Sculpt addresses this gap by providing a centralized platform where all visual feedback is organized by project, version, and discussion thread, ensuring that every piece of feedback is traceable and actionable.

The platform supports project-based organization with role-aware access control, enabling teams to manage who can view, comment on, or modify design assets. Its version management system allows designers to upload iterative revisions of their work, while reviewers can compare versions side-by-side and track how feedback has been incorporated over time.

Key capabilities of Sculpt include:
\begin{enumerate}[label=\Roman*.]
    \item Providing a browser-based collaborative review environment accessible from any device.
    \item Supporting image upload with automatic version tracking for iterative design workflows.
    \item Enabling real-time visual annotations using drawing tools directly on design assets.
    \item Facilitating structured threaded discussions with reply, like, and resolve capabilities.
    \item Implementing role-based project membership with invite links for team collaboration.
\end{enumerate}

\subsection{1.2\quad Feature Proposal}

\subsubsection{1.2.1 Project-Based Organization:}

Implementing a project-based organizational structure will serve as the foundation for managing design assets and team collaboration. Users can create projects, invite collaborators with specific roles (owner, editor, viewer), and organize their visual assets within dedicated project workspaces. This structure ensures that feedback and discussions remain contextually organized and easily navigable.

\subsubsection{1.2.2 Image Upload and Version Management:}

Adding robust image upload and version management capabilities will enable designers to maintain a complete history of their design iterations. Each uploaded image supports multiple versions, allowing teams to track the evolution of design decisions. Version-specific previews and metadata ensure that reviewers always reference the correct iteration during feedback sessions.

\subsubsection{1.2.3 Visual Annotation Tools:}

Integrating a canvas-based annotation system within the review interface will facilitate precise visual feedback. Reviewers can use drawing tools including pencil, line, and rectangle instruments with customizable colors to mark specific areas of concern directly on design assets. An undo/redo history stack supports iterative annotation refinement before submission.

\subsubsection{1.2.4 Structured Commenting System:}

Developing a threaded commenting system will enable organized and contextual discussions on design assets. Comments can be linked to specific visual annotations, creating a direct connection between written feedback and the marked area on the design. Features such as replies, likes, and resolve toggles support a complete review workflow from issue identification to resolution.

\subsubsection{1.2.5 Share Links and Collaborative Access:}

Introducing token-based share links will simplify the process of onboarding new collaborators to projects. Project owners can generate invitation links that allow external team members to join with appropriate permissions, streamlining the collaboration setup process without requiring manual account provisioning by administrators.

\section{2 Literature Survey / Related Works}

In the domain of collaborative design tools, the need for effective visual feedback mechanisms has driven significant innovation in web-based review platforms. Modern design workflows increasingly rely on real-time collaboration features that enable distributed teams to communicate feedback efficiently. Technologies such as WebSocket communication, canvas-based rendering, and RESTful API architectures form the backbone of these collaborative systems.

\subsection{2.1\quad Real-time Collaboration in Web Applications:}

Real-time collaboration in web applications has been extensively explored through technologies like WebSockets and server-sent events. Frameworks such as Socket.IO enable bidirectional communication between clients and servers, facilitating instant updates across connected users. In the context of design review platforms, real-time capabilities ensure that annotations, comments, and status changes are immediately visible to all team members, reducing feedback latency and improving iteration speed [1].

\subsection{2.2\quad Canvas-Based Annotation Systems:}

The HTML5 Canvas API provides a powerful foundation for building interactive annotation tools within web applications. Research and industry practice demonstrate that canvas-based drawing interfaces can be effectively combined with image rendering to create overlay annotation systems. Libraries and frameworks built on the Canvas API enable features such as freehand drawing, geometric shape placement, and color-coded annotations, all essential for precise visual feedback in design review contexts [2].

\subsection{2.3\quad Design Review and Version Control Platforms:}

Existing design review platforms such as Figma, InVision, and Marvel have established patterns for collaborative design feedback. These tools demonstrate the value of version-controlled design assets, threaded comment discussions, and annotation-linked feedback. However, many existing solutions are tightly coupled to specific design tools or require proprietary file formats, creating barriers for teams that work across multiple design applications [3].

\subsection{2.4\quad Role-Based Access Control in Collaborative Systems:}

Implementing fine-grained access control in collaborative web applications is critical for maintaining data integrity and user trust. Research on role-based access control (RBAC) models demonstrates effective patterns for managing permissions across multi-user platforms. JWT-based authentication combined with server-side role verification provides a scalable approach to securing collaborative features while maintaining session efficiency [4].

In summary, combining modern web technologies --- including Next.js for frontend rendering, Express.js for API services, Prisma ORM for database management, and HTML5 Canvas for annotation --- provides a comprehensive technical foundation for building a collaborative visual feedback platform that addresses the gaps in existing design review workflows.

\section{3 Problem Definition}

In contemporary product development environments, design review and visual feedback processes remain fragmented and inefficient. Teams working on visual assets --- including UI designs, marketing materials, architectural plans, and creative content --- frequently rely on a combination of email threads, messaging applications, standalone screenshot annotation tools, and verbal communication to exchange feedback. This scattered approach creates several critical challenges that impede productivity and collaboration quality.

The primary issues include the lack of version tracking for design iterations, making it difficult to determine which version of an asset is being discussed or whether previous feedback has been addressed. Additionally, existing feedback mechanisms often disconnect written comments from the specific visual area they reference, forcing reviewers to describe locations verbally (e.g., ``the button in the top-right corner'') rather than pointing directly at the element in question.

Furthermore, access control in shared review workflows is frequently inadequate. Many teams resort to sharing files via cloud storage links with broad access permissions, risking unauthorized modifications or data exposure. The absence of structured permission models means that feedback from different stakeholders (designers, developers, managers, clients) cannot be appropriately scoped.

The identified limitations also include:
\begin{itemize}
    \item Lack of centralized project-based organization for visual assets and their associated feedback.
    \item Absence of annotation tools that directly link visual markup to discussion threads.
    \item No version management system to track design evolution and feedback incorporation.
    \item Limited role-based access control for managing reviewer permissions within shared projects.
    \item Cumbersome onboarding processes for adding new collaborators to active review sessions.
\end{itemize}

To address these challenges, Sculpt aims to provide a unified web-based platform that combines project organization, version-tracked image management, canvas-based annotation tools, and structured commenting into a single cohesive review workflow. The platform is designed for professionals, design teams, and organizations that require an efficient, centralized solution for collaborative visual feedback.

Success will be measured through the completeness of the implemented feature set, the reliability of real-time collaboration features, the effectiveness of the annotation-to-comment linking workflow, and the platform's ability to support multi-user review sessions with appropriate access controls.

\section{4 Objectives}

\begin{itemize}
    \item To design and implement a secure authentication system with JWT-based session management, supporting both standard user and administrative access flows within the Sculpt platform.

    \item To develop a project management module with full CRUD operations, role-based membership, and token-based invitation links for seamless team collaboration.

    \item To build an image upload and version management system that supports iterative design workflows, enabling users to upload, rename, delete, and version visual assets within organized project workspaces.

\end{itemize}

These objectives aim to deliver a comprehensive visual feedback platform that addresses the fragmentation in current design review workflows. By combining project organization, version management, visual annotation, and structured discussion into a unified interface, Sculpt seeks to reduce feedback latency, improve review clarity, and support efficient collaboration for distributed design teams.

\section{5 Methodology}

A systematic and iterative approach has been employed to achieve the outlined objectives, involving sequential development sprints with design, implementation, testing, and integration phases. The following methods are used:

\subsection{5.1\quad Process Model}

\textbf{Requirement Gathering:} In this stage, the requirements and specifications for the collaborative visual feedback platform are collected from stakeholders including design teams, reviewers, and development team members. This stage is critical in understanding the project scope, goals, and constraints.

\textbf{Design:} In this stage, the system architecture and software design are created based on the requirements and specifications. The design includes the frontend framework (Next.js), backend API framework (Express.js), database schema (Prisma ORM), file storage strategy, and authentication considerations.

\textbf{Development:} Implement the designed features using appropriate technologies and frameworks, including Next.js for frontend development, Express.js for backend API development, Prisma for database operations, and HTML5 Canvas API for annotation rendering. Follow best practices and coding standards to ensure the application's scalability, maintainability, and security.

\textbf{Testing:} Conduct thorough testing of each feature to identify and resolve any defects or inconsistencies. The testing includes functional, integration, and security testing through both manual and automated testing methods. Each biweekly sprint concludes with validation of the completed milestone.

\textbf{Integration:} Integrate the developed features into the existing monorepo infrastructure, ensuring seamless interoperability and functionality between the frontend (apps/web) and backend (apps/api) modules. Perform end-to-end testing to verify the overall system behavior and user experience.

\textbf{Deployment:} In this stage, the application is prepared for deployment on cloud infrastructure. The deployment involves setting up the server, database, file storage, and configuring the application for production use.

\textbf{Maintenance:} In this stage, the application is monitored and maintained to ensure it continues to function correctly and meet the changing needs of end-users. This involves bug fixes, updates, and enhancements based on feedback and usage data.

The Sculpt project follows an iterative and incremental approach, where each biweekly sprint builds on the previous one and feedback is incorporated throughout the development cycle. This model allows for flexibility and agility in adapting to changing requirements and user needs.

\subsection{5.2\quad Modules}

These module components provide the user interface and functionality for the Sculpt collaborative visual feedback platform.

\begin{itemize}
    \item[\textbf{$\triangleright$}] \textbf{Authentication Module:}
    \begin{itemize}
        \item Register: Allows new users to create an account with email and password.
        \item Login: Enables users to sign in to their existing account with JWT token issuance.
        \item Logout: Clears session cookies and invalidates the current session.
        \item Profile: Retrieves authenticated user information for session restoration.
        \item Admin Auth: Provides a separate authentication path for administrative users.
        \item Auth Context: Manages client-side authentication state and route protection.
    \end{itemize}

    \item[\textbf{$\triangleright$}] \textbf{Project Management Module:}
    \begin{itemize}
        \item CreateProject: Enables users to create new project workspaces via a modal interface.
        \item ProjectList: Displays all projects accessible to the authenticated user on the dashboard.
        \item ProjectCard: Renders individual project entries with actions and preview information.
        \item ProjectSettings: Allows in-place editing of project metadata through update endpoints.
        \item DeleteProject: Implements delete confirmation and state cleanup workflows.
        \item Membership: Manages role-based access control (owner, editor, viewer) for projects.
    \end{itemize}

    \item[\textbf{$\triangleright$}] \textbf{Collaboration Module:}
    \begin{itemize}
        \item InviteFlow: Handles invite-to-project endpoint integration for adding collaborators.
        \item MembersModal: Provides UI for managing member invite and removal actions.
        \item ShareLinks: Implements token-based share link creation, listing, and revocation.
        \item JoinPage: Connects join token pages to share acceptance endpoints for onboarding.
        \item RoleHandling: Wires role display and permission-aware controls throughout the UI.
    \end{itemize}

    \item[\textbf{$\triangleright$}] \textbf{Image and Version Module:}
    \begin{itemize}
        \item ImageUpload: Handles multipart file upload with automatic first-version creation.
        \item ImageListing: Displays uploaded files tied to each project with preview thumbnails.
        \item ImageDetail: Provides detailed retrieval flow for individual image review.
        \item VersionUpload: Supports adding new versions to existing images for iterative updates.
        \item FileOperations: Implements rename, delete, and preview URL assembly for assets.
    \end{itemize}

    \item[\textbf{$\triangleright$}] \textbf{Annotation Module:}
    \begin{itemize}
        \item CanvasCore: Implements the annotation canvas drawing engine with coordinate-based rendering.
        \item DrawingTools: Provides pencil, line, and rectangle tool behaviors with tool-state switching.
        \item ColorPicker: Enables color selection in the annotation toolbar for visual categorization.
        \item HistoryStack: Manages undo/redo controls for iterative annotation corrections.
        \item AnnotationState: Handles annotation arrays and canvas refresh flow consistency.
    \end{itemize}

    \item[\textbf{$\triangleright$}] \textbf{Commenting Module:}
    \begin{itemize}
        \item CommentCRUD: Implements create, read, update, and delete operations for comments.
        \item Replies: Supports parent-child threaded conversations on specific issues.
        \item Likes: Provides like/unlike toggle functionality with per-user reaction tracking.
        \item ResolveToggle: Enables resolved state management for issue closure tracking.
        \item CommentSidebar: Renders nested replies, metadata, and search/filter capabilities.
        \item AnnotationLink: Links comment annotation data to canvas highlight actions.
    \end{itemize}
\end{itemize}

These modules collectively contribute to the functionality and user experience of the Sculpt platform, enabling seamless visual feedback and collaboration during design review sessions.

\section{6 Work Done So Far}

\subsection{6.1\quad Project Setup and Environment Configuration}

The initial phase of development focused on establishing the foundational infrastructure for the Sculpt platform. A monorepo structure was set up with separate modules for the frontend (apps/web) and backend (apps/api). The development environment was configured with Node.js, pnpm as the package manager, and local development scripts. The Express.js server was bootstrapped with base middleware, and initial route groups were mounted for authentication, projects, images, comments, and administration. The Prisma ORM schema was defined with core entities including User, Project, Image, and Membership models, and initial database migrations were applied.

On the frontend, the Next.js application shell was set up with route layouts and base navigation paths. Reusable UI components and modal primitives were integrated to standardize the interface across pages. Shared TypeScript interfaces were added to reduce payload mismatches between the API and frontend.

\textit{Figure 3: Project Dashboard Interface}

\begin{itemize}
    \item The monorepo structure separates frontend and backend concerns for maintainability.

\subsection{6.2\quad Authentication System}

A complete authentication system was implemented spanning both backend and frontend. The backend includes register, login, and logout controller logic with JWT token issuance and HTTP-only cookie handling. Authentication middleware secures all protected routes by validating JWT tokens on each request. A separate admin authentication path was implemented with isolated token flow.

On the frontend, login and register pages were connected to live API responses with proper success and failure state handling. An AuthContext was implemented for session state management and protected route redirection. Credentialed fetch patterns were established globally to maintain cookie-based session consistency. Route guards enforce unauthenticated access redirection, and comprehensive endpoint testing validated all authentication paths.

The collaboration system includes an invite-to-project flow with a members management UI for invite and removal actions. Token-based share links allow project owners to generate invitation URLs that external collaborators can use to join projects. Server-side permission checks prevent unauthorized updates and deletions, ensuring data integrity across all project operations.

The frontend integrates an image upload modal within the project view, and the project content view displays uploaded files with preview thumbnails generated from the latest available version. File operations including rename and delete are supported, with frontend validation handling edge cases such as empty selection and invalid request states.

A color selection toolbar enables visual categorization of feedback marks.

An undo/redo history stack allows reviewers to make iterative corrections while annotating. The annotation state management system keeps drawing data consistent between actions and handles canvas refresh and reset operations after comment submission. The annotation footer connects the drawing payload with comment submission, ensuring that visual markup is stored alongside written feedback.
